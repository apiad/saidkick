# 0.5.0 — AXTree + opt-in console + highlight refcount + shadow DOM

**Status:** approved per the 2026-04-21 hardening roadmap.
**Scope:** 5 items bundled because they all touch content.js / locator resolution / background-side CDP.

## 1. `by_role` locator via CDP AXTree

Architecture: **role-matching is background-side, then rewritten to a CSS selector for content-script dispatch.** AXTree access requires `chrome.debugger`; content.js doesn't speak CDP. By resolving role-matches to a unique CSS selector in the background, all other locator logic (ambiguity detection, `waitForLocator`, `--within-css`) stays in content.js untouched.

Flow for any command when `by_role` is set:
1. Background receives the frame with `by_role` populated.
2. `ensureDebuggerAttached(tabId)` (no-op if already).
3. `Accessibility.getFullAXTree` — flat array of AX nodes.
4. Filter nodes where `role.value === by_role`. If `by_text` is also set, filter further by accessible name (substring-ci / exact / regex per the usual escape hatches).
5. For each surviving match, `DOM.resolveNode({ backendNodeId })` → get `RemoteObject.objectId`.
6. `Runtime.callFunctionOn` with a tiny function that walks the node up to `<body>` producing a unique `:nth-of-type` CSS path (same logic as `uniqueSelector` in content.js, but computed background-side).
7. Rewrite the payload: drop `by_role` + `by_text`; set `css` to the computed selector. Forward to content.js via the existing path.

Edge cases:
- AXTree has no match → reply with `"element not found"` (routes to 404).
- Multiple matches and caller didn't pass `nth` → reply with `"Ambiguous locator: found N matches"` (400). Caller can retry with `--nth`.
- If `Accessibility.getFullAXTree` fails (rare — tab crashed) → 502 upstream.

CLI: `saidkick find --tab T --by-role button --by-text "Send"` etc., added to every selector-using command. Server-side: `Locator` mixin gains `by_role: Optional[str] = None`.

## 2. Opt-in console mirroring

Today `main_world.js` auto-installs on every page and mirrors every console call to the server, which violates "the user's session is only partly under saidkick's control."

Architecture: **main_world.js still auto-installs (document_start), but postMessage-forwarding through content.js is gated on a per-tab mirror flag stored in `chrome.storage.session`.** Page-level wrapping is unavoidable (main_world has no chrome.* APIs) but the log never leaves the page unless opt-in.

New endpoints and commands:
- `POST /mirror` body: `{tab: "br-XXXX:N", enabled: bool}`. Server sends `SET_MIRROR` to the extension; extension writes `chrome.storage.session.set({ [`mirror_${tabId}`]: enabled })`.
- `GET /mirror?tab=...` — returns the current flag.
- CLI: `saidkick mirror on --tab TAB` / `saidkick mirror off --tab TAB` / `saidkick mirror status --tab TAB`.

Content.js change: at postMessage receipt for `saidkick-log`, do a `chrome.storage.session.get(`mirror_${tabId}`)` — but content.js doesn't know its own tabId directly. Instead, background stamps the state via `chrome.tabs.sendMessage` when it changes; content.js keeps a local `let mirroring = false;` and toggles on those messages. Simpler than storage round-trip per log.

Also: cover `console.info` and `console.debug` while we're redoing main_world.

**Breaking:** default is now `mirror=false`. Callers who relied on auto-mirrored logs must opt in per tab. Called out prominently in CHANGELOG.

## 3. Highlight prev-style via WeakMap + refcount

Back-to-back highlights on the same element pollute the saved `prev` styles. Fix:

```javascript
const highlightState = new WeakMap();  // Element -> {prev, activeCount}

// on HIGHLIGHT:
let entry = highlightState.get(el);
if (!entry) {
    entry = { prev: captureStyles(el), activeCount: 0 };
    highlightState.set(el, entry);
}
entry.activeCount += 1;
applyHighlight(el, color);
if (duration > 0) setTimeout(() => {
    entry.activeCount -= 1;
    if (entry.activeCount === 0) {
        restore(el, entry.prev);
        highlightState.delete(el);
    }
}, duration);
```

Prev is captured on first call, restored only when refcount returns to zero. Concurrent highlights overlay; the last one to finish cleans up.

## 4. Shadow DOM-aware locators

Add `pierce_shadow: bool = False` (default off for back-compat). When true, `collectLocator`'s text/label/placeholder scan descends into shadow roots recursively:

```javascript
function* walkDeep(root) {
    yield root;
    const all = root.querySelectorAll("*");
    for (const el of all) {
        yield el;
        if (el.shadowRoot) yield* walkDeep(el.shadowRoot);
    }
}
// then: [...walkDeep(root)].filter(pred)
```

CSS selectors don't natively pierce (`querySelectorAll` stops at shadow boundaries), so for `--pierce-shadow --css X` we walk all shadow roots and call `querySelectorAll(X)` on each.

Perf: pierce is off by default. When on, we pay a tree-walk — fine on most sites, potentially slow on pages with many thousands of nodes + shadow roots (rare).

## 5. `ensureDebuggerAttached` error propagation

Today `Page.enable` / `Runtime.enable` fail silently:

```javascript
await new Promise(r => chrome.debugger.sendCommand(target, "Page.enable", {}, r));
```

The callback ignores `chrome.runtime.lastError`. Fix each to reject on error (same pattern as `Input.dispatchKeyEvent`):

```javascript
await new Promise((resolve, reject) =>
    chrome.debugger.sendCommand(target, "Page.enable", {}, () =>
        chrome.runtime.lastError ? reject(chrome.runtime.lastError) : resolve()
    )
);
```

| Situation | HTTP | Message |
|---|---|---|
| `by_role` with no AXTree match | 404 | `"element not found"` |
| `by_role` ambiguous + no `nth` | 400 | `"Ambiguous locator: found N matches"` |
| `Accessibility.getFullAXTree` failed | 502 | bubbled from CDP |
| `POST /mirror` missing `enabled` | 422 | pydantic |
